最后介绍下 webpack.dll.config.js 配置。
在使用webpack进行打包时候，对于依赖的第三方库，比如vue，vuex等这些不会修改的依赖，我们可以让它和我们自己编写的代码分开打包，这样做的好处是每次更改我本地代码的文件的时候，webpack只需要打包我项目本身的文件代码，而不会再去编译第三方库，那么第三方库在第一次打包的时候只打包一次，以后只要我们不升级第三方包的时候，那么webpack就不会对这些库去打包，这样的可以快速的提高打包的速度。因此为了解决这个问题，DllPlugin 和 DllReferencePlugin插件就产生了。webpack.dll.config.js 就是用来配置 DllPlugin 的。

DLLPlugin 这个插件把所有的第三方库依赖打包到一个bundle里面，还会生成一个 bundle-manifest.json文件。bundle-manifest.json文件是一个第三方库的映射。
DllReferencePlugin 这个插件是在webpack.prod.config.js中使用的，该插件的作用是将webpack.dll.config.js打包生成的budle.dll.js文件引用到需要的预编译的依赖上来。bundle-manifest.json文件会包含所有库代码的一个索引，当在使用webpack.prod.config.js配置打包文件时，会使用DllReferencePlugin插件读取bundle-manifest.json文件，看看是否有该第三方库。

总结来说，第一次使用 webpack.dll.config.js 文件会对第三方库打包，打包完成后就不会再打包它了，然后每次运行 webpack.prod.config.js配置打包时，只会打包项目自身的代码，当需要使用第三方依赖的时候，会使用 DllReferencePlugin插件去读取第三方依赖库。所以说它的打包速度会得到一个很大的提升。

this指向

Posted on 2018-11-15 | In JS/TS

this指向是很基础也是老生常谈的话题了。不过时间久了也是容易遗忘，出现混淆的情况。故在这里做一个记录吧，方便之后查看回忆。

ES5的函数中的this 对象是在运行时基于函数的执 行环境绑定的，而箭头函数没有自己的 this 绑定。箭头函数中使用的 this，其实是直接包含它的那个非箭头函数或函数表达式中的 this

直接调用

直接调用，就是通过 函数名(…) 这种方式调用，执行环境是全局对象。这时候，函数内部的 this 指向全局对象，在浏览器中全局对象是 window，在 NodeJs 中全局对象是 global。

注意： 严格模式下，this为 undefined;

bind/call/apply 调用

这个三个方法可以传入参数修改函数中的this，修改了执行环境

注意：如果函数本身通过bind绑定了 this ，使用 apply 和 call 的时候，传入的对象不会生效

作为对象的方法调用

这种情况下，函数中的 this 指向调用该方法的对象，因为此时的执行环境是对应的对象。

如果现将函数方法赋值给某个变量，然后通过变量调用的话，则this取决于变量调用的环境，而非对象

此时t的执行环境是全局对象，所以this指向全局对象

new 调用

而在 es5 中，用 new 调用一个构造函数，会创建一个新对象，而其中的 this 就指向这个新对象

匿名函数中的this

匿名函数的执行环境具有全局性，因此其 this 对象通常指向 全局对象

箭头函数

前面说过箭头函数没有自己的 this 绑定。箭头函数中使用的 this，其实是直接包含它的那个非箭头函数或函数表达式中的 this

object.getNameFunc 包含着箭头函数，并且通过对象方法的形式调用，故其中的this指向对象 object,所以箭头函数中的this也指向 object。

object.getNameFunc 包含着箭头函数，而object.getNameFunc赋值给了变量foo，foo()在全局环境中执行，故其中的this指向全局对象,所以箭头函数中的this也指向全局对象。

CDN工作原理

CDN是通过接管DNS查询来工作的，具体工作原理如下：

  1. 用户输入访问的域名,操作系统向 LocalDns 查询域名的ip地址.
  2. LocalDns向 ROOT DNS 查询域名的授权服务器(这里假设LocalDns缓存过期).
  3. ROOT DNS将域名授权dns记录回应给 LocalDns.
  4. LocalDns得到域名的授权dns记录后,继续向域名授权dns查询域名的ip地址.
  5. 域名授权dns 查询域名记录后(一般是CNAME)，回应给 LocalDns.
  6. LocalDns 得到域名记录后,向智能调度DNS查询域名的ip地址.
  7. 智能调度DNS 根据一定的算法和策略(比如静态拓扑，容量等),将最适合的CDN节点ip地址回应给 LocalDns.
  8. LocalDns 将得到的域名ip地址，回应给 用户端.
  9. 用户得到域名ip地址后，访问站点服务器
  10. CDN节点服务器应答请求，将内容返回给客户端.(缓存服务器一方面在本地进行保存，以备以后使用，二方面把获取的数据返回给客户端，完成数据服务过程)

如何判断是否命中CDN

一般来说,CDN供应商会在响应头中加X-Cache之类的字段，以腾讯CDN为例，会使用X-Cache-Lookup响应头来表示：
X-Cache-Lookup:Hit From MemCache 表示命中CDN节点的内存
X-Cache-Lookup:Hit From Disktank 表示命中CDN节点的磁盘
X-Cache-Lookup:Hit From Upstream 表示没有命中CDN
